## Answered inline (no session needed)

**"I don't understand `TYPE_META`/`REST` in `graph_html.py`"** —
`TYPE_META` (`pipeline/exports/graph_html.py:40`) is the node legend: one
entry per `rg:`/`skos:` class rendered in the graph, each with a colour (from
a colour-vision-deficiency-checked 8-slot palette) and a size (the required
*second* channel, per the project's CVD-accessibility rule — colour alone
isn't a safe encoding). `REST` (line 309) is unrelated to `TYPE_META`: it's
the force-directed layout's per-edge-kind rest length (the "natural" spring
distance the canvas physics settles toward) — `broader` edges pull skills
close to their category hub (55px), `evidencedBy` edges are looser (95px),
so the layout visually clusters by relationship type without you coding
cluster positions by hand.

**"`_CSS` block in `pdf.py`"** — WeasyPrint (the PDF renderer) takes CSS as
either a file path or a plain string; there's no templating step, so the
stylesheet lives as one triple-quoted Python string (`pdf.py:179`) rather
than an external `.css` file. Same pattern as `graph_html.py`'s `_TEMPLATE`.
Worth extracting both to real files — see WS4.

**"Is jsonresume.org the JSON-LD output?"** — No, two separate exports:
`dist/graph.jsonld` is the framed JSON-LD *of the graph itself* (used by the
site + SPARQL panel + `<head>` structured data). `dist/resume.json` is a
*different* schema — the jsonresume.org resume schema, built by
`pipeline/exports/json_resume.py` for ATS/resume-tool compatibility. Same
source graph, two independent target schemas.

**Certification-date schema gap ("I don't understand this question")** — the
confusion is fair; here's the mechanism. `context.jsonld` is a lookup table:
frontmatter key → RDF predicate. If a cert note's frontmatter has `issued:
2020-01`, but no context entry maps `issued` to a predicate, the vendored
exporter (SPEC-compliant behavior) silently skips that key with a warning —
the date stays sitting in the Markdown file but never becomes a triple, so
nothing downstream (SHACL, SPARQL, the graph render, a future date filter)
can see it. The decision was never abstract — your later items (date
granularity, education dates, cloud-azure) already answered it: yes, wire
the keys in. WS1 is that work.

## Work mode: sessions and model tiers

Group by **workstream**, not by raw `next-prompts.md` line — most lines are
sub-details of one edit. Tier by **ambiguity and blast radius**, not by
file-count:

- **Haiku** — mechanical, fully-specified edits: adding notes from a given
  template, renaming per a worked example, backfilling a property across
  many files once its schema exists. High volume, near-zero judgment calls.
- **Sonnet** — the default. Schema edits (new predicate + shape + context
  wiring), source-document research with a judgment call at the end, pipeline
  code changes to existing structure, diagram synthesis.
- **Opus** — cross-cutting reasoning with real cost if wrong: the work-project
  migration (touches schema + 9 projects + every skill's evidence + bullet
  quality), the date-axis/sidebar feature (open architecture question baked
  into the ask), anything going into the *private* application repo (a live
  job-application artifact, worth the best model).

Respect the dependency chain below — a session run out of order will hit a
missing predicate or a shape that rejects its own output.
