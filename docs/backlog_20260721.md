## Context established 2026-07-21 (research, not yet decided)

**Minting and resolving are independent.** RDF requires only that a subject
be a valid IRI. Nothing must answer an HTTP `GET` on it. `https://` is chosen
because DNS already solved global uniqueness — resolvability is a separate,
optional courtesy (Linked Data principle 3).

**What the exporter mints today** (`pipeline/vendor/vault_to_rdf.py:510`,
`minted_iri`): governing `@base` + `quote(stem, safe="")`. The governing base
is the note's own ontology/vocabulary `context.jsonld` for `_schema/` notes,
otherwise the nearest `context.jsonld` walking up, ending at
`vault/context.jsonld` → `https://joseph-higaki.github.io/resume-graph/id/`.
Folders never enter the IRI (SPEC §4.5); location travels as `vld:path`.
Result: 222 instance subjects, spelled `data:AWS%20S3`, `data:AI%20Metis%20Engineer`.

Hash vs slash is already split correctly: `rg:` and the category scheme use
**hash** namespaces (`…/vocab/rg#level`) so one document can describe every
term; instances use a **slash** namespace, so each needs its own document to
resolve.

**Current resolution state — everything 404s.** Probed live 2026-07-21:

```
https://joseph-higaki.github.io/resume-graph/       200 text/html   (only index.html is deployed)
https://joseph-higaki.github.io/resume-graph/id/    404
https://joseph-higaki.github.io/resume-graph/graph.ttl    404
https://joseph-higaki.github.io/resume-graph/graph.jsonld 404
```

**GitHub Pages serving constraints** — verified empirically against live
Pages sites, not assumed:

| deployed as | served as |
|---|---|
| extensionless file | `application/octet-stream` ✗ |
| `foo.ttl` | `text/turtle; charset=utf-8` ✓ |
| `foo.jsonld` | `application/ld+json` ✓ |
| directory `foo/` + `index.html` | `301` → `foo/` → `text/html` ✓ |

Pages is a static file server: it **ignores `Accept`** (no content
negotiation) and cannot issue `303 See Other`. One representation per path,
selected by path alone. That rules out the textbook httpRange-14 dance and
forces the choice in WS13.

### WS12 — Decision + policy: IRI stability (`@id` pinning)
**Tier: Opus (decision) then Sonnet (mechanical) · Depends on: none · Domain: schema + data**

Identity is currently derived from a mutable filename. Two live exposures:

1. **Rename churn.** Cool URIs don't change, but WS3's planned Udacity
   renames (`Udacity DE — Songplay Postgres Schema` →
   `Songplay — Python ETL for Postgres star schema`) *change those projects'
   identities*. Cost is zero today — nothing external cites them — and stops
   being zero the moment the site is public and indexed. **Order matters:
   decide this before WS3 runs, or accept the churn knowingly.**
2. **Filename namespace is flat and global.** Two notes with the same stem in
   different folders mint the *same* IRI and silently merge. Already hit once
   (`Data Engineering Zoomcamp Coursework` renamed to dodge the cert note).
3. **Host lock-in.** `joseph-higaki.github.io` is a GitHub *username*. Rename
   the account or move to a custom domain and all 222 IRIs die. Mitigations:
   a custom domain, or `w3id.org` permanent redirects fronting Pages
   (community-run, added by PR — verify current process before committing).

The lever exists already: `subject_iri` (`vault_to_rdf.py:517`) lets a note
override its IRI with an absolute `@id:` in frontmatter, decoupling identity
from label while the filename stays human-friendly.

- [ ] **Decide the pinning scope.** Standing recommendation from the
      2026-07-21 session: pin `@id` on the entity kinds most likely to be
      renamed or cited — projects, positions, certs — and leave skills
      filename-derived, since a skill's `prefLabel` effectively *is* its
      identity and renaming one is a modeling change anyway. Alternatives:
      pin everything (maximally stable, 222 extra lines, IRIs stop being
      readable), or pin nothing (status quo, accept churn).
- [ ] **Decide the host question separately** — custom domain vs `w3id.org`
      vs accept `github.io`. Cheap now, expensive after publication.
- [ ] Once scoped: add `@id:` to the chosen notes, and a SHACL check (or a
      pytest) that every note in the pinned classes carries one, so the policy
      can't silently erode.
- [ ] Record both decisions in an ADR (`docs/adr/0002-iri-minting-and-resolution.md`).

### WS13 — Courtesy dereferencing: per-entity pages under `/id/`
**Tier: Sonnet · Depends on: WS12 (don't publish IRIs you're about to change) · Domain: mechanism**

Make the minted IRIs answer. Not required for correctness — required because
*this project is the portfolio artifact*, and "my resume entities are
dereferenceable and return schema.org JSON-LD" is a demonstrable claim in a
way "I picked a URL-shaped string" is not. It is also what makes the graph
federatable: someone else can `owl:sameAs` a node and their tooling can
follow it.

Given the Pages constraints above, the workable pattern is **HTML-first with
embedded JSON-LD** — one representation that serves both audiences and
sidesteps the missing content negotiation entirely. Already implied by the
project brief ("embed the JSON-LD in `<head>` as schema.org structured data").

- [ ] New export `pipeline/exports/id_pages.py`, reading `dist/graph.ttl`
      (never the vault — requirement 7), wired into `make export`. Output is
      derived and disposable.
- [ ] One directory per subject: `dist/id/<decoded stem>/index.html`, with a
      **literal** space in the directory name — the server percent-decodes the
      request path, so `GET /id/AWS%20S3` → `301` → `…/AWS%20S3/` → that file.
      Page carries the human view (prose excerpt, level, evidence links) plus
      `<script type="application/ld+json">` with the entity's triples.
- [ ] Optional pure-RDF path: sibling `<stem>.ttl` / `<stem>.jsonld`, linked
      via `<link rel="alternate" type="text/turtle">`. Pages types both
      correctly (table above), so this costs only the emit.
- [ ] Same treatment for the vocabularies: `dist/vocab/rg/index.html` and
      `dist/vocab/categories/index.html`, so `…/vocab/rg#level` lands on a
      page with that term as an anchor. One document each — the whole point of
      the hash namespace.
- [ ] Deploy `dist/graph.ttl` and `dist/graph.jsonld` at the site root while
      here; both currently 404 and the SPARQL panel (M5) will need them.
- [ ] Test: every subject in `dist/graph.ttl` has a corresponding page, and no
      page exists without a subject.
