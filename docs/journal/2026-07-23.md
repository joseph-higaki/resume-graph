## Decisions (and why)
- **Three diagrams, not one.** The repo has three lifecycles with different
  triggers and contracts; one merged sequence (~15 participants) would bury the
  two properties the diagrams exist to show — the gate *ordering* in `make all`
  and the repo *trust boundary* in publishing.
- **Real symbols, not abstractions.** Every message names the actual function
  (`export_graph()`, `_prune_unevidenced_claims()`, …) so the doc is falsifiable
  against the code: a renamed stage makes the diagram visibly stale instead of
  vaguely right.
- **Scope = as-built only.** M4/M5 site and M7/M8 Neo4j are stated as
  out-of-scope in the doc header rather than drawn aspirationally — a diagram
  with nothing to check it against invites drift.
- **Mermaid now, Lucid later** (per WS9's own framing). Diagram 3 gains the most
  from a Lucid port (repo swimlanes as containers); 1–2 port roughly 1:1.
